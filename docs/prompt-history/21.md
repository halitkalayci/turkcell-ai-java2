## product-service — Governance Uyum Raporu

---

### [KRİTİK] 1 — V2 Liste Endpoint'i Contract İhlali (AGENTS.MD §2.1)

**Kural:** OpenAPI doğruluğun tek kaynağıdır. Implementation sözleşmeye uymalıdır.

**Sorun:** ProductPage.java içindeki `items` alanı `List<ProductResponse>` tipini kullanıyor. `ProductResponse` V1 DTO'sudur; `description` alanı **yoktur**. V2 OpenAPI kontratı (product-service-v2.yml) `ProductPage.items` elemanlarının V2 `Product` şemasına (zorunlu `description` alanıyla) uygun olmasını şart koşar.

**Etkilenen dosyalar:**

| Dosya | Hata |
|---|---|
| ProductPage.java | `List<ProductResponse>` → `List<Object>` (generic değil), V2 response tipi dönülemez |
| ProductServiceV2.java | `list()` metodu `toV1Response()` çağırıyor, V2 response üretilmiyor |
| ProductControllerV2.java | `listProducts()` dönüş tipi `ResponseEntity<ProductPage>` — içindeki item tipi yanlış |

**Beklenen:** V2 için `ProductPageV2` (üreten items tipi `List<ProductResponseV2>`) ayrıca oluşturulmalı.

---

### [KRİTİK] 2 — V1 Create/Update Response Şeması Hatalı (AGENTS.MD §2.1)

**Sorun:** V1 OpenAPI `POST /api/v1/products` ve `PUT /api/v1/products/{id}` response şeması `Product` olarak tanımlanmış (product-service-v1.yml). Implementation ise `ProductResponse` dönderiyor. İki schema'da ortak alanlar (`id, name, price, stock, sku`) aynı olsa da farklı şema isimlerinden kaynaklanabilecek entegrasyon sorunlarına zemin hazırlanıyor. Bu, teknik olarak field uyuşmazlığı değil, isimlendirme uyuşmazlığıdır — annotation'sız Swagger UI'da yanlış schema gösterir.

---

### [ORTA] 4 — Fonksiyon Soyutlama Seviyesi İhlali (02-clean-code-functions §3)

**Kural:** Bir fonksiyon içindeki tüm ifadeler aynı soyutlama seviyesinde olmalıdır.

**updateStock() — her iki serviste:**

```java
// YANLIS - save + field access tek satırda, iki soyutlama seviyesi karışıyor
return new StockUpdateResponse(product.getId(), product.getName(), productRepository.save(product).getStock());
```

**create() ve replace() — her iki serviste:**

```java
// YANLIS - low-level if/throw doğrudan yüksek seviye iş mantığının içinde
public ProductResponse create(ProductCreateRequest request) {
    if (productRepository.existsBySku(request.sku())) { // düşük seviye detay
        throw new SkuAlreadyExistsException(request.sku());
    }
    Product product = new Product(...); // nesne inşası
    return toResponse(productRepository.save(product)); // save + map tek satırda
}
```

`validateSkuUniqueness()`, `buildProduct()` gibi private metodlara ayrılmalıdır.

---

### [ORTA] 5 — Do One Thing İhlali (02-clean-code-functions §2)

`create()` ve `replace()` metodları şu işlemleri tek method içinde yapıyor: validation + domain object build + persist + response mapping. Her adım bağımsız private metoda ayrılmalıdır.

---

### [DÜŞÜK] 6 — Kullanılmayan Repository (AGENTS.MD §1.1 - uydurma yok)

CategoryRepository.java tanımlı ve inject edilebilir durumda ancak hiçbir controller/service tarafından kullanılmıyor. `Category` entity'si `Product` ile `@ManyToOne` ilişkisinde ama kategori hiçbir API endpoint'inde yönetilmiyor. Dead code durumunda.

---

### [DÜŞÜK] 7 — V1 ve V2 Spec'inde Kullanılmayan Schema

Her iki OpenAPI dosyasında da `ProductPatchRequest` şeması `components/schemas` altında tanımlı ancak hiçbir path tarafından referans edilmiyor. PATCH endpoint implementasyonu ya şema silinmeli ya da endpoint eklenmeli.
